## 평가 대상 기능들

이 서비스의 핵심 기술 요소는 크게 다섯 가지다:

1. AI 페르소나 — 장기 기억 + 일관된 성격 유지
2. 멀티 에이전트 세계 — 캐릭터끼리 서로 사는 세계 시뮬레이션
3. 지연 발송 & 먼저 연락하기 — 사람처럼 느껴지는 응답 패턴
4. 감정 상태 관리 — 화나면 떠나기, 친해지면 반말 전환
5. 부가 기능 — 음성 통화, 사진 피드백, 적응형 난이도

## 기능별 구현 난이도

### ✅ 쉬운 것들 (기존 기술 그대로 가져다 쓰면 됨)

**지연 발송 & 먼저 연락하기**

AI 문제가 아니라 **표준 백엔드 엔지니어링**이다.
"30분 후에 이 메시지 보내"는 Redis + Celery 같은 작업 큐(Job Queue)로 처리한다.
"먼저 연락할지 판단"은 스케줄러가 주기적으로 "이 캐릭터가 연락할 타이밍인가?" 평가하고 LLM에게 메시지 생성을 요청하면 된다.

> **Job Queue**: 해야 할 일을 목록에 넣어두고 순서대로 처리하는 시스템.
> 배달 앱이 "30분 후 도착 예정" 알림을 보내는 방식과 같은 원리.

**음성 통화 / 캐릭터별 목소리**

ElevenLabs라는 서비스가 정확히 이 용도로 만들어졌다.
캐릭터별 목소리를 설계해서 API에 등록해두면, 텍스트를 그 목소리로 읽어준다.
OpenAI TTS API도 있고, 선택지가 많다. 캐릭터마다 다른 목소리 배정은 어렵지 않다.
실시간 통화 시 약간의 지연(latency)이 아직 있지만, 기술이 빠르게 좋아지고 있다.

**사진 찍어서 피드백 받기**

GPT-4V, Claude Vision, Gemini Vision 모두 이미지를 이해하고 설명하는 기능이 있다.
유저가 거리 간판이나 메뉴판을 찍으면, 선생님 페르소나가 "이건 ~라는 뜻이에요"로 답하는 건 API 연결 수준의 작업이다.

**감정 상태 관리 (화나면 떠나기, 친해지면 반말 전환)**

RPG 게임의 호감도 시스템과 구조가 같다.
DB에 각 캐릭터의 상태를 숫자로 저장하고, AI한테 컨텍스트로 전달한다.

```
// 예시: DB에 저장되는 캐릭터 상태
{
  characterId: "sujin",
  closeness: 72,      // 친밀도 (0-100)
  irritation: 30,     // 짜증 수치 (0-100)
  speechStyle: "formal" // formal → casual 전환 기준
}
```

AI는 이 상태를 프롬프트로 받아서 그에 맞게 반응한다.
"closeness < 20이면 대화방 퇴장 이벤트 트리거" 같은 규칙은 게임 로직이다.

### ⚠️ 어려운 것들 (구현 가능하지만 공을 많이 들여야 함)

**장기 기억 일관성**

대화가 몇 달 쌓이면 문제가 생긴다.
LLM(대형 언어 모델)은 한 번에 처리할 수 있는 텍스트 양(컨텍스트 창)이 제한돼 있어서,
6개월 전 대화를 그대로 다 넣을 수가 없다.

> **컨텍스트 창 (Context Window)**: LLM이 한 번에 읽고 기억할 수 있는 텍스트의 최대 길이.
> 사람으로 치면 "단기 기억 용량"에 해당한다.

해법은 **RAG + 벡터 DB** 방식이다.
모든 대화를 벡터로 변환해서 저장해두고, 현재 대화와 관련 있는 기억만 꺼내서 AI한테 전달한다.
"수진이가 이전에 내 고양이 얘기를 들은 적 있다" 같은 기억을 검색해서 꺼낼 수 있다.

> **RAG (Retrieval-Augmented Generation)**: "검색 후 생성" 방식.
> AI가 모든 걸 외우는 게 아니라, 필요할 때 관련 정보를 검색해서 가져와 답하는 구조.
> 구글 검색을 달고 다니는 AI라고 생각하면 된다.
>
> **벡터 DB**: 텍스트를 숫자 배열(벡터)로 변환해서 저장하는 데이터베이스.
> "의미가 비슷한 문장끼리 가까이 저장"되기 때문에 "관련된 기억"을 빠르게 찾을 수 있다.

완벽하진 않지만 "그럴싸하게 기억하는" 수준은 충분히 가능하다.
Mem0 같은 라이브러리가 이미 이 구조를 구현해두었다.

**멀티 에이전트 세계 시뮬레이션 (캐릭터끼리 사는 세계)**

이게 이 서비스에서 가장 어려운 부분이다.
"민지가 수연이랑 카페 갔다가 싸웠대" 같은 가십을 만들려면,
캐릭터들이 유저 없이도 서로 상호작용하고 이벤트를 생성해야 한다.

**근데 이게 이미 증명된 기술이다.**

2023년 스탠퍼드 대학에서 "Generative Agents"라는 논문을 발표했다.
AI 마을에 에이전트 25명을 넣고 각자 일상을 보내게 했다.
아침에 일어나고, 직장 가고, 친구 만나고, 파티 계획하고, 소문을 퍼뜨렸다.
실제로 의도하지 않은 사회적 행동(약속 잡기, 관계 형성)이 자연스럽게 나타났다.

> 논문 링크: [Generative Agents: Interactive Simulacra of Human Behavior (2023)](https://arxiv.org/abs/2304.03442)
> 이 논문이 Korean World 세계관의 기술적 기반이 되는 레퍼런스다.

다만 **비용 문제**가 있다. 에이전트 20명이 서로 상호작용하면 LLM 호출이 많아진다.
해법은 **배치 시뮬레이션** — 실시간으로 돌리는 게 아니라
"오늘 세계에서 일어난 일"을 하루 1-2번 배치로 처리하면 비용 통제 가능하다.

> **배치 처리 (Batch Processing)**: 작업을 실시간으로 하나씩 처리하는 게 아니라,
> 모아두었다가 한 번에 처리하는 방식. 비용과 서버 부하를 크게 줄일 수 있다.

## 진짜 어려운 문제 — 비용 구조

기술적 한계보다 **경제적 한계**가 더 현실적인 벽이다.

헤비 유저 한 명이 하루 30분 사용한다고 가정하면:

| 항목 | 예상 월 비용 |
|---|---|
| LLM API (대화 생성) | $3–8 |
| TTS 음성 생성 | $1–3 |
| 에이전트 시뮬레이션 | $1–2 |
| **합계 (유저 1인당)** | **$5–13** |

$9.99 구독료에서 서버비·호스팅까지 빼면 마진이 거의 없거나 적자다.

이건 해결 불가능한 문제가 아니다. 세 가지 방향이 있다:

1. **가격 인상**: 창립 멤버 $9.99 이후 정가 $19.99, 장기적으로 더 높게
2. **사용량 제한**: 하루 N회 메시지, 음성 통화 월 M분 등 티어 설계
3. **모델 최적화**: 초기엔 비싼 GPT-4 사용, 이후 파인튜닝된 작은 모델로 전환

> **파인튜닝 (Fine-tuning)**: 기존 AI 모델을 특정 용도에 맞게 추가 학습시키는 것.
> 이 앱의 대화 패턴에 맞게 훈련하면 더 저렴한 모델로도 비슷한 품질이 나올 수 있다.

초기 단계(유저 100명)에서는 신경 안 써도 된다. 월 API 비용 $500–1,300 수준이라 감당 가능.
유저가 1,000명을 넘어갈 때부터 비용 최적화를 진지하게 고민해야 한다.

## "AI 같은 느낌"이 나는 문제에 대해

유저가 끊임없이 메시지를 보내면 AI 패턴이 드러날 수 있다는 우려가 있는데,
이 앱의 구조 자체가 그 문제를 완화하도록 설계되어 있다:

- **지연 응답**: 즉각 답장이 없으니 "처리 속도"가 보이지 않는다
- **먼저 연락**: AI가 필요할 때만 연락 → 패턴 노출 빈도 줄어듦
- **대화 거부·무시**: "지금 바빠요"가 오히려 사람처럼 느껴지게 함
- **카카오톡 스타일**: 띄엄띄엄 주고받는 구조가 AI의 약점(긴 대화 시 패턴 노출)을 자연스럽게 가려줌

가장 리스크가 큰 상황은 유저가 AI에게 "너 AI지?"라고 직접 물어볼 때다.
이 상황에 대한 응답 전략(페르소나 유지 vs. 솔직하게 인정)을 초기에 설계해두어야 한다.
