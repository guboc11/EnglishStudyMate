# Operating Model (MVP Delivery)

## Roles
- Planner (AI): 스펙/우선순위/의존성 정리, 작업 티켓 작성·업데이트, 스케줄 관리.
- Developer (human, senior): 기술 의사결정, 티켓 수행/분배, 코드 품질 책임, 리뷰 승인.
- Workers (AI, 2명): 주니어 개발자 역할로 병렬 구현/리팩토링/테스트 수행.

## 기본 흐름
1) Planner가 목표·의존성을 분석해 작업 티켓을 작성하고 순서를 잡는다.
2) Developer가 티켓을 선택해 수행 계획을 확정하고, 병렬 가능한 작업을 Workers에 분배한다.
3) Workers가 부분 구현/리팩토링/테스트를 완료하면 Developer가 통합·리뷰·수정한다.
4) 각 사이클 종료 시 Planner에 진행 현황·리스크를 공유하고 필요 시 보드 재정렬을 요청한다.

## 운영 원칙
- 작업 보드 기반: 모든 작업은 티켓으로 추적하고 의존성은 Planner가 명시한다.
- 병렬화 우선: 충돌 없는 작업은 Workers에 분배해 리드타임을 단축한다.
- 짧은 사이클: 한 사이클마다 결과물을 통합해 회귀 리스크를 최소화한다.
- 로그·결정 기록: 주요 결정과 차단 요소는 티켓 코멘트나 관련 문서에 즉시 기록한다.

## 산출물/위치
- 계획/스펙: `Execution/Plans/` 및 Product docs.
- 진행 현황: 작업 보드(외부 도구) + 티켓 ID를 커밋/노트에 병기.
- 코드/데이터: `Product/` 하위. 초기에는 파일 기반 mock, 이후 API 연동.

## 실행 요청 규칙
- npm/pnpm install 등 의존성 설치가 권한/네트워크 문제로 로컬에서 실패하면 즉시 사용자에게 실행 요청한다 (Planner/Developer/Workers 모두 동일 적용).
- 승인/실행 요청 전에는 설치를 우회하려 하지 않는다.
- 설치가 필요해 보이면 사전 안내 후 사용자 확인을 받고 진행한다.
